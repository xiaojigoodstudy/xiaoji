# 00 - 总体架构总览

## 1. 目标与边界

- 目标：建设一个偏个人使用、三端兼容（小程序/公众号/App）的日常工具集平台。
- 一期范围：`信息雷达`、`自动提醒`、`打卡`。
- 非目标（一期不做）：商业计费、复杂社交关系、多租户隔离策略落地。

## 2. 总体架构

- 架构风格：`模块化单体 + 事件化`。
- 客户端：微信小程序、公众号 H5、移动 App、管理端 Web、管理端 H5。
- 服务端：
  - `Gateway`：统一入口、鉴权、限流、审计。
  - `BFF-C`：聚合 C 端接口。
  - `BFF-Admin`：聚合管理端接口。
  - `Core Service`：领域模块（IAM/Radar/Rule/Notification/Tool-Habit）。
  - `XXL-JOB`：抓取与定时任务调度。
  - `MQ`：跨模块异步事件。

## 3. 一期核心业务链路

1. 雷达抓取：源配置 -> 定时任务 -> 解析去重 -> 入库检索。
2. 规则触发：新内容事件 -> 规则引擎判定 -> 触发提醒事件。
3. 消息触达：提醒事件 -> 通道适配（微信/App）-> 发送日志与重试。
4. 行动闭环：用户收到提醒 -> 进入事项/打卡 -> 日统计回流。

## 4. 模块职责边界

- `iam`：身份认证、角色权限、会话令牌。
- `radar`：信息源、抓取记录、内容解析、标签分类。
- `rule-engine`：规则定义、条件表达式、命中判定。
- `notification`：模板、通道、发送日志、重试。
- `tool-habit`：事项、打卡、统计。
- `platform`：系统配置、字典、审计日志。

## 5. 关键设计原则

- 单一职责：每个模块只处理单一业务域。
- 边界隔离：模块间仅通过服务接口或事件交互，禁止跨模块直查表。
- 可扩展：新工具以独立模块接入，不改动现有核心流程。
- 可观测：所有关键链路具备日志、指标、告警、追踪。

## 6. MVP 验收标准

- 可配置至少 1 个雷达信息源并稳定抓取。
- 规则命中后 1 分钟内完成至少 1 个渠道触达。
- 用户可创建打卡事项并完成连续打卡统计。
